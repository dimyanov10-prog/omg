

# Банковский беспредел — Writeup

**Категория:** форензика  
  

## Описание

Когда то давно, в прекрасные времена, когда Копатыч был ещё молод и не страдал от геморроя и артрита, он жил в РССС — Ромашковом Советском Союзе Смешариков. В те неспокойные 90 е он отправил другу перевод с важным сообщением, но оно потерялось в суматохе и в грохоте валящихся столбов электропередач.
У нас тут чисто случайно оказался дамп трафика того периода. Помоги Копатычу вспомнить, а то, судя по всему, у него ещё и Альцгеймер начал развиваться, и память подводит всё сильнее.


## файл:

`capture.pcap`


## Осмотр pcap

Сначала посмотрел общую информацию о дампе:

``` bash
capinfos capture.pcap
```

Стало понятно, что это не обычный web‑трафик. Wireshark не показывает
привычных TCP‑стримов.

Тип канала оказался `LIN`, поэтому стандартные фильтры вроде
`tcp.stream` здесь бесполезны. Пришлось разбирать payload вручную.

## Структура пакетиков

После просмотра нескольких пакетов стало видно, что кадры выглядят
примерно так:

    7e .... 7e

`0x7e` используется как начало и конец кадра.

Пятый байт выглядит как тип сообщения. Если разделить пакеты по этому
байту, получается несколько потоков.

Текстовые сообщения:

    0x15
    0x05
    0x04
    0x02

Бинарный поток:

    0x1e

Ещё один поток:

    0x01

Он оказался просто текстом на русском, каким‑то рассказом.

## Извлечение payload

Сначала крч я думал, что полезная часть фиксированной длины. Но оказалось,
что если брать только 16 байт, часть данных теряется.

Правильнее извлекать payload так:

``` python
payload = d[6:-3]
```

## Банковские сообщения

Если собрать текстовые потоки (`0x15`, `0x05`, `0x04`, `0x02`),
получается структура очень похожая на старые SWIFT‑сообщения.

Пример сообщения:

    {1:F01...}
    {2:I100...}
    {4:
    :20:...
    :32A:...
    :50:...
    :59:...
    :57A:...
    :70:...
    -}

Поле `:70:` это описание платежа.

Иногда в конце строки появляется странный суффикс:

    .XX

где `XX` это hex‑байт.

Я подумал не просто так сюда будут пихать такое. Если собрать все такие суффиксы подряд, получается длинная hex‑строка.

пароль.

## Поиск пароля

Берём скрытую последовательность и пробуем с ней преобразования:

``` python
hidden_hex = (
      "6fe4aee4a4e4a9e4a4e4af14e4a3e4a4e49de4ace49de4a1e4a96914"
      "e4aee48ae480e48ae486e48ae48114e5b5e48fe48ae486e48a141914"
      "7760724f044c52526b5204415a506b5907490c021e474a47060c140e"
      "080a06380b12140f060e38243321"
  )

  data = bytes.fromhex(hidden_hex[:-2])

  for op in ("xor", "add", "sub"):
      for k in range(256):
          if op == "xor":
              out = bytes(b ^ k for b in data)
          elif op == "add":
              out = bytes((b + k) & 0xFF for b in data)
          else:
              out = bytes((b - k) & 0xFF for b in data)

          s = "".join(chr(b) if 32 <= b < 127 else "." for b in out)
          print(op, hex(k), s)
```

из всех вариантов получилась строка сработала на  XOR с 0x67 (SIX SEVEN !!!!!!!!!!!!)
