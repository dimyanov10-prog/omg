# Банковский беспредел (forensics) — writeup

## TL;DR

- Из `capture.pcap` удалось выделить несколько LIN-потоков.
- В одном из потоков (`type=0x1e`) оказался ZIP с картинками и шифрованным `flag/flag.txt`.
- Пароль к `flag.txt` спрятан в другом скрытом блоке (из суффиксов платежных сообщений), после `xor 0x67`.
- Полученная строка-пароль: `aksioma_lushai_CTF`.
- Флаг: `axiom{x.25_15_th3_b3st_pr0t0c01}`.

---

## Дано

Файл: `capture.pcap`.

По легенде — старый банковский трафик, в котором потерялось важное сообщение.

## 1. Первичный осмотр pcap

Проверка информации о дампе:

```bash
capinfos capture.pcap
```

Из важного:

- много пакетов;
- нестандартный для классического web/network CTF тип канала (LIN);
- значит нужен ручной разбор payload, а не обычный `tcp.stream`.

## 2. Разбор кадров и выделение полезных потоков

Кадры имели вид:

- начало/конец: `0x7e ... 0x7e`;
- внутри управляющие поля и payload.

Быстро выяснилось, что по байту `type` (`d[4]`) идут разные логические потоки:

- `0x15`, `0x05`, `0x04`, `0x02` — текстовые блоки с банковскими MT-подобными сообщениями;
- `0x1e` — бинарный поток (в итоге это ZIP);
- `0x01` — отдельный текстовый кусок (русский рассказ).

Ключевой момент: payload кадра не фиксированной длины.  
Нужно брать `d[6:-3]` (а не только 16 байт из 25-байтных кадров), иначе теряется хвост.

## 3. Что было в текстовых банковских потоках

В потоках `0x15/0x05/0x04/0x02` восстанавливаются сообщения вида:

```text
{1:F01...}
{2:I100...}
{4:
:20:...
:32A:...
:50:...
:59:...
:57A:...
:70:...
-}
```

В поле `:70:` иногда есть суффикс `.XX` (hex-байт).  
Из этих суффиксов формируется скрытая последовательность.

Именно после корректной сборки (с переменной длиной payload) получилась полная скрытая hex-строка:

```text
6fe4aee4a4e4a9e4a4e4af14e4a3e4a4e49de4ace49de4a1e4a96914e4aee48ae480e48ae486e48ae48114e5b5e48fe48ae486e48a1419147760724f044c52526b5204415a506b5907490c021e474a47060c140e080a06380b12140f060e38243321
```

## 4. ZIP-поток и шифрованный flag

Из `type=0x1e` собирается ZIP (local headers):

- `picture1.jpg ... picture22.jpg`;
- `flag/flag.txt` (AES encrypted, method 99).

То есть флаг уже рядом, но нужен пароль.

## 5. Где оказался пароль

Берем скрытую последовательность из шага 3 и пробуем простую побайтную операцию:

```python
decoded = bytes(x ^ 0x67 for x in hidden_bytes)
```

В результате появляется читаемая строка:

```text
aksioma_lushai_CTF
```

Это и есть пароль к `flag/flag.txt`.

## 6. Расшифровка `flag/flag.txt`

Проверка пароля проходит успешно (PBKDF2-HMAC-SHA1 параметры WinZip AES).

После AES-CTR расшифровки + deflate получаем:

```text
axiom{x.25_15_th3_b3st_pr0t0c01}
```

## Флаг

```text
axiom{x.25_15_th3_b3st_pr0t0c01}
```

## Мини-скрипт воспроизведения идеи (сокращённо)

```python
import struct, re

# 1) читаем pcap, собираем payload по type (d[4]) из кадров 7e ... 7e
# payload = d[6:-3]

# 2) из type=0x15 парсим сообщения и берем .XX из конца :70:
# hidden_hex = ''.join(xx)

# 3) password = xor(hidden_bytes, 0x67) -> b"aksioma_lushai_CTF"

# 4) открываем AES zip entry flag/flag.txt этим паролем
# 5) получаем флаг
```
